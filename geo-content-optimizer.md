  与我们规划的 Skill 的关系

  现有 Skill 覆盖了什么： 内容差距分析（对应原框架中"检测内容是否被 AI 引用"的思路，但更实战——直接拿 Google AI Overview
  做对比）

  我们规划的 Skill 覆盖了什么： Schema 生成、可读性检测、集群规划、可见度监控、季度报告

  结论： 两者完全互补。建议在技术方案中增加 模块 E（内容差距分析），直接复用 geo-content-optimizer 作为子流程。

  方案调整建议

  在原有 4 模块基础上新增：

  ┌─────────────────┬───────────────────────────────────────────────┬───────────────────────────────────────┐
  │      模块       │                     说明                      │               实现方式                │
  ├─────────────────┼───────────────────────────────────────────────┼───────────────────────────────────────┤
  │ E. 内容差距分析 │ 输入 URL → 对比 AI Overview → 找缺口 → 出建议 │ 直接调用 /geo-content-optimizer <url> │
  └─────────────────┴───────────────────────────────────────────────┴───────────────────────────────────────┘

  在 SKILL.md 中加入引导逻辑：
  - 当用户提供 URL 要求优化时 → 调用 geo-content-optimizer
  - 当用户要求生成 Schema/检测/规划时 → 走我们自己的模块

  这样最终的 Skill 变成一个 "GEO 全流程中枢"：自身负责 Schema 生成、可读性检测、集群规划、效果监控，同时调度
  geo-content-optimizer 做内容差距分析。
